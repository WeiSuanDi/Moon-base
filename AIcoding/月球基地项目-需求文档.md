## 一、项目是什么

一个关于"月球基地建设构想"的**交互式网页产品**，部署在 Vercel 上。

核心体验：用户进入网站看到一颗可交互的 3D 月球，月球表面漂浮着若干真实命名地点的点选光标。用户点击一个地点作为基地选址，然后通过一步步的决策（能源、水源、辐射防护等）搭建出一套属于自己的基地配置，每个选择都会带来连锁后果。走完流程后，可以生成一份只属于这套配置的产出（可行性简报 / 海报文案 / 叙事）。

**它首先是一件"作品 / 游戏"，AI（agent）是其中关键位置的增强，不是主体。**

---

## 二、最重要的架构原则（违反这条，整个项目就跑偏了）

判断一段逻辑该不该用 LLM（agent），只问一句话：

> **"如果把 LLM 删掉，这部分会不会坏掉？"**

- **会坏 → 用 agent**（开放式推理、生成式产出，规则表写不出来的东西）
- **不会坏 → 用确定性代码**（菜单、数值计算、状态更新、渲染，写死更快更稳更便宜）

由此得出的物理分工，必须体现在代码结构里：

| 层 | 跑在哪 | 负责什么 | 删掉 LLM 会怎样 |
|---|---|---|---|
| **前端（游戏本身）** | 用户浏览器 | 3D 月球、点选、选项菜单、数值后果计算、状态管理、UI 渲染 | 照常运行 |
| **后端（agent 能力）** | Vercel Serverless Function，仅被调用时运行 | 两件事：① 用户的开放式追问 ② 收尾的生成式产出 | 这两处会失效 |

**agent 不是"一个聊天窗口"，是"前端在需要时调用的一个能力/引擎"。** 它的输出有时是一段对话文本，有时是一份生成的产出，由前端决定怎么用、怎么展示。

## 四、关键文件职责

### `scripts/state.js` —— 整个项目除 agent 外最关键的文件

它是一个普通的 JS 对象，是**单一状态源（single source of truth）**。月球点选、菜单选择，改的都是它；它一改，UI 跟着重渲染；它也是每次请求后端时塞进请求体里的东西。

v1 状态结构：

```js
const baseState = {
  site: null,        // 选址，如 "shackleton"
  energy: null,      // "nuclear" | "storage" | "solar"
  water: null,       // "isru" | "earth_supply"
  radiation: null,   // "regolith" | "cave" | "hull"
  history: []        // 用户的选择轨迹，按顺序记录，用于收尾产出
};
```

### `api/main.py` —— FastAPI app

Vercel 现在（2026）支持以 Beta 形式直接运行 FastAPI（ASGI）应用，入口文件可命名为 `main.py`。挂两个路由：

- `POST /api/agent` —— 开放式追问。请求体 `{ state, question }`，返回 `{ answer }`。
- `POST /api/summary` —— 收尾产出。请求体 `{ state, history }`，返回 `{ result }`。

**关键约束：Serverless Function 不记得上一次调用（每次冷启动是全新实例）。** 所以后端是**无状态的**——它不保存任何会话记忆，前端每次请求都必须把完整的 `baseState` 带过来，后端只基于这次收到的 state 作答。这一条决定了整个数据流，务必遵守。

DeepSeek 调用配置：

```python
# 用 openai 库，指向 DeepSeek
client = OpenAI(
    api_key=os.environ["DEEPSEEK_API_KEY"],
    base_url="https://api.deepseek.com"
)
# model 用 "deepseek-chat"
```

### `vercel.json`

告诉 Vercel：根目录的静态文件直接发，`/api` 下走 Python runtime。
**请按 Vercel 官方 Python runtime 文档的最小示例来写，不要凭空猜格式**——这块 2026 年仍是 Beta，以官方最新示例为准。

### `.env.local` 与密钥安全（务必做到）

- `.gitignore` 必须包含 `.env.local`，**绝不能把它提交到 GitHub**，否则 API key 泄露。
- key 真正生效的地方是 **Vercel 后台的环境变量设置**（部署后在网站上手动填 `DEEPSEEK_API_KEY`）。
- `.env.local` 仅供本地测试用。

## 五、v1 范围与实现顺序（严格分阶段，不要一口气全做）

### v1 只做一条完整的路：**沙克尔顿环形山 + 三步建设**

不要一上来铺四个地点、五个决策。先把一条路从头到尾打通、能演示，加点、加步以后都只是复制扩展。

### 实现顺序（三个阶段，每阶段独立可见可验证）

**阶段 1：纯前端确定性游戏（完全不接 agent）**
1. 3D 月球渲染 + 沙克尔顿这一个点选光标，可点击。
2. 点击 → 往 `state.site` 写入 `"shackleton"` → UI 高亮选中点。
3. 三步建设决策，每步是一个选项菜单（见第六节），用户选择 → 写入 state → 前端**用规则表算出数值后果**（能源缺口、水冰量、辐射剂量等）并展示 → 更新配置侧栏。
4. 此阶段全部跑在浏览器里，删掉 LLM 照样能玩。**先让游戏能完整玩一遍。**

**阶段 2：打通前后端链路（agent 先不做真功能）**
1. 创建 `api/main.py`，一个最简单的 FastAPI app。
2. 接通 DeepSeek，前端 `agent-client.js` 能 fetch 到回复。
3. **哪怕这个路由只会原样返回"收到"，也先验证：git push 之后，前端真的能调用到这个函数。** 先证明链路通。

**阶段 3：把 agent 真正用在那两个点上**
1. `/api/agent`：开放式追问——把当前 state 放进 system prompt，结合领域知识回答用户跳出菜单的自由提问。
2. `/api/summary`：收尾产出——拿完整 state + history 生成一份属于这套配置的产出。

### v1 明确**不做**的事
- 不做数据库 / 持久化存储（状态只活在前端 JS 变量里）。
- 不做用户登录 / 账号。
- 不做"保存基地 / 分享链接"（以后再加存储时做）。
- agent 追问**先不接真正的向量检索 RAG**——v1 用一段静态的、塞进 system prompt 的领域知识片段顶替即可。**不要在 Serverless Function 里去搭向量数据库**（冷启动和托管都是坑），真 RAG 留作 v2 增强。

## 六、沙克尔顿这条路的具体内容

### 选址：沙克尔顿环形山（南极）
特点：永久阴影坑内有水冰，环形山口几乎永昼。主打"能源 + 水"近乎最优，代价是极端低温、地形险峻。后续三步的后果推演都要结合这个选址的特点。

### 三步建设决策（沙盘型：摆出选项 + 各自后果，用户拍板，前端把后果算给他看）

1. **能源**：核能 / 大规模储能 / 太阳能
2. **水源**：就地采水冰（ISRU）/ 地球补给
3. **辐射防护**：埋入月壤 / 利用洞穴 / 加厚舱壁

每步前端要做：结合选址说明该决策的约束、摆出选项、用户选完用规则表算出后果并写进 state。数值规则表可以先用合理的近似值（占位），后续再校准——重点是先把"选择→后果→状态更新"这个闭环跑通。

> 交互气质为**沙盘型**（用户自己拍板，agent/前端把"你这么选会怎样"算给你看），不是顾问型（agent 直接替你做主）。

## 七、一个待你（项目负责人）拍板的开放决策

`/api/summary` 的产出形式还没最终确定，有三种候选：
- **基地可行性简报**（偏严肃、专业）
- **任务招募海报文案**（偏视觉、传播）
- **"假如你是首批居民"的叙事片段**（偏故事、情感）

v1 可先默认做**基地可行性简报**（最直接），它只是 `/api/summary` 里 prompt 的一个设计，后续随时可换。但注意一个连带关系：**产出需要什么信息，建设过程就得收集什么信息**——所以这个决策最终会反向影响三步建设要问出哪些东西。
